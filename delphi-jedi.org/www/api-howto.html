What is an API Header?

An API (Application Programming Interface) is a collection of programming libraries distributed as a single unit along with the interfacing necessary to allow applications to access these units. On Windows, APIs most frequently they take the form of a number of DLLs. The headers are C source modules that publish the declarations and symbols for the various function calls exposed to the interface when the particular DLL is loaded.

Function headers for the libraries are coded in a consistent, format that supplies an interface for programmers to access application features from their own applications. API headers are almost always written in C or C++. Project JEDI's objective is to convert these headers into Pascal, providing pre-translated interfaces for Delphi programmers.

We refer to these "Pascal versions" as conversions or translations. The two terms are interchangeable. However, because the two languages are not, there's often a lot of refactoring involved in converting (translating) the headers.

Project Jedi and Borland

When Project JEDI began in 1997, our main purpose was to write interface units for APIs that were not supported in Delphi releases. The project's dedication to that goal of converting APIs for free use by any Delphi developer remains.

In the first quarter of 1999, Borland approached Project JEDI, proposing a close liaison between our participants and their own Delphi product development effort. In March that year, Borland formally announced the relationship, its support for our work and its intention to incorporate our best products in the next Delphi release.

These days the relationship is described by Borland [Codegear] Developer Support as The JEDI Initiative and you can read up on it HERE.

Standards and Guidelines

Two overlapping levels of standards compliance exist: Project JEDI standards (which largely deal with how translations should be packaged) and Borland standards (which are concerned with source code layout and methodology).

Borland supplied some tools and the set of strict guidelines that needs to be followed to make our work compliant with Delphi R&D standards. For copyright and version control reasons, Borland would not allow us provide our own downloadable kits of the standards and the supporting tools. Unfortunately, some of these resources have since been moved or lost.

If you find a dead link, by all means post a message to the support list for the API conversions.

The 'JEDI-certified' marks The 'Borland-compliant' marks

      

      

JEDI programmers are encouraged to produce Delphi code that conforms to the set of code and conversion guidelines that was established with the publication of the Project JEDI Conversion Tutorial; and to the pack specifications defined in How to Present Your API Conversion. Conforming work will be granted JEDI Certification once its developers have completed the testing and documentation requirements.

The Borland requirements differ a little as to the naming and use of some code elements and objects. To be eligible to be considered for inclusion with the next release of Delphi, the work must comply with the guidelines published in these two documents:

Pascal Header Translation Specification
Borland Code Style Guide

Non-Standard Code

Non-standard donated conversions are perfectly acceptable. Donors will be assisted to adapt their code to standard if they wish.
